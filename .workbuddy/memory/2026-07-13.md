## Built HHeader — MV3 request-header editor extension
- Manifest V3 Chrome extension to customize / modify / delete HTTP request headers per domain, ModHeader-style Profiles.
- Stack: Vue 3 + SCSS popup, Vite multi-entry build, declarativeNetRequest dynamic rules, chrome.storage.local persistence.
- Key files: src/core/{constants,storage,ruleEngine}.js, src/background/service-worker.js, src/popup/{App.vue, composables/useProfiles.js, components/*}.
- vite.config: `root: 'src'` so popup HTML -> dist/popup/, background -> dist/background/; `base: './'` for relative assets under chrome-extension://.
- Rule engine: Add->append, Modify->set, Filter->remove. Wildcard domain -> `||host` urlFilter; regex -> regexFilter (validated via isRegexSupported; unsupported ones skipped).
- append DOWNGRADE (CORRECTED): Chrome allows `append` ONLY on a protected allowlist (user-agent/cookie/accept-*/x-forwarded-for...). So Add keeps `append` on allowlisted headers; on any other (non-allowlist) header it downgrades to `set`, otherwise Chrome rejects the rule.
- Confirmed regexFilter + modifyHeaders works in current Chrome (no webNavigation fallback needed).
- Build verified (dist/ correct). Rule engine unit-tested headlessly with mocked chrome.

## Fixed "Rule with id 1 does not have a unique ID" runtime error
- CAUSE: `service-worker.js` called `updateDynamicRules` with `removeRuleIds: currentRuleIds` and `addRules: rules` in the SAME call. Chrome requires these two ID sets to be disjoint. Since ruleEngine always allocates IDs starting from 1, new `rules` (e.g. `[1,2,3,4]`) overlapped with the previously stored `currentRuleIds` (e.g. `[1,2,3]`), triggering the error. Also, `currentRuleIds` is an in-memory array that resets to `[]` on service-worker restart, so it could not reliably clear stale rules.
- FIX: in `syncRules()`, first fetch `getDynamicRules()` to obtain the ACTUAL current dynamic rules, remove them with `addRules: []`, then add the new set in a second `updateDynamicRules()` call. Recovery path unchanged. Build green; unit tests 62/62. Error should disappear after reloading the extension.
